## Обоснование идеи

Современные научные исследования в областях STEM (математика, физика, химия, биология, информатика) всё чаще требуют написания программного кода для:

* численного моделирования
* анализа данных
* проверки гипотез
* воспроизводимости экспериментов

При этом значительная часть времени исследователя уходит не на саму научную гипотезу, а на:

* формализацию задачи
* декомпозицию решения
* написание вспомогательного кода

Предлагаемая система представляет собой мультиагентную LLM-архитектуру, которая принимает научную задачу в естественном языке и автоматически формализует, декомпозирует, генерирует и тестирует код решения.

## Цель

Создать агентную систему, которая ускоряет научный процесс за счёт автоматизации этапов:

* формализации задачи
* алгоритмической декомпозиции
* генерации и тестирования экспериментального кода

## Метрики

**Продуктовые метрики:**

* Research time reduction — сокращение времени от формулировки задачи до готового кода

**Агентные метрики:**

* Качество декомпозиции задачи (см. раздел ниже)
* Улучшение pass@1 при использовании декомпозиции vs. прямая генерация
* Стабильность workflow — доля pipeline без system-level ошибок

**Технические метрики:**

* pass@1 на тестовых задачах
* p95 latency на полный workflow
* Доля runtime-ошибок

### Как измеряется качество декомпозиции

Качество декомпозиции измеряется через три метрики:

1. **Execution success rate** — доля запросов, где сгенерированный на основе декомпозиции код успешно выполнился без runtime-ошибок
2. **Test pass rate** — доля тест-кейсов, прошедших на сгенерированном коде (`success_rate` из TestingAgent)
3. **Improvement over baseline** — сравнение pass@1 между:
   - *прямой генерацией* (задача → код без декомпозиции)
   - *декомпозиция + генерация* (задача → план → код)

Если `pass@1(с декомпозицией) > pass@1(без декомпозиции)` — декомпозиция добавляет ценность.

## Потенциальные сценарии использования

### 1. Алгоритмические и математические задачи

Исследователь формулирует задачу:

> Given an integer x, determine whether it is a palindrome.

Система декомпозирует, предлагает алгоритм, генерирует код.

### 2. Генетика и биология

> Скрещивание двух гетерозиготных растений гороха (Aa × Aa).

Система формализует модель наследования, строит решётку Пеннета, пишет код для расчёта распределений фенотипов.

### 3. Химия

> Определить относительную молекулярную массу H₂SO₄.

Система извлекает атомные массы, строит вычислительную модель, генерирует код для автоматического расчёта.

### 4. Физическое моделирование

> Три одинаковых заряда расположены в вершинах равностороннего треугольника.

Система формализует задачу через закон Кулона, выводит систему уравнений, пишет код для численного решения.

### 5. Численное моделирование (ODE/симуляции)

> Simulate Lotka-Volterra predator-prey system with prey=40, predators=9, alpha=0.1, beta=0.02.

Система строит план (математическая модель → ODE → scipy.integrate), генерирует ~50 строк кода.

### Edge cases

| Сценарий | Ожидаемое поведение |
|----------|---------------------|
| Задача без формальной постановки («сделай что-то умное») | Декомпозиция возвращает уточняющий план; код может быть нерелевантным |
| Противоречивые условия («найти все простые числа больше бесконечности») | Генерация кода с некорректным выводом; тесты не пройдут |
| Задача требует внешних данных (конкретный CSV-датасет) | Код генерируется с заглушками; runtime-ошибка при выполнении |
| Задача выходит за context window LLM | Декомпозиция или код обрезаются; SyntaxError в тестах |
| Задача с небезопасными требованиями (удалить файлы, HTTP-запрос) | AST-проверка блокирует выполнение; test_results содержит ошибку безопасности |
| Задача на языке отличном от русского/английского | Система работает, но качество декомпозиции может снизиться |
| Недостаточное описание («оптимизируй это» без кода) | Decomposer генерирует общий план; код будет шаблонным |
| Задача требует нескольких файлов/модулей | Генерируется один файл; модульность нарушена |

## Ограничения

### Latency

| Этап | Время |
|------|-------|
| Декомпозиция | 3–7 с |
| Генерация кода | 10–20 с |
| Тестирование | 5–15 с |
| Оптимизация (1 итерация) | 10–20 с |
| Форматирование | 3–7 с |

p95 latency ≈ 40–70 секунд (с 1–2 итерациями оптимизации)

### Надёжность

Основные проблемы:
* Галлюцинации LLM (неверный алгоритм или синтаксис)
* Несоответствие формата тест-кейсов и кода (mismatch stdin-формата)
* Зависимость от доступности LLM API

### Операционные ограничения

* Зависимость от Celery + RabbitMQ + Redis при развёртывании
* Локальная модель (Ollama) не требует интернета, но ограничена ресурсами хоста

## Архитектурный набросок

Система состоит из пяти специализированных агентов и одного оркестратора:

1. **AgentRouter** — центральный оркестратор; управляет последовательностью и переходами
2. **DecomposerAgent** — декомпозиция задачи в пошаговый план
3. **CodeGeneratorAgent** — генерация Python-кода по задаче и плану
4. **TestingAgent** — генерация тест-кейсов и выполнение кода через subprocess
5. **OptimizationAgent** — анализ ошибок тестов и исправление кода
6. **FormatterAgent** — финальное форматирование ответа для пользователя

**Используемые интеграции:**
- LangChain + OpenAI-compatible API (LLM)
- subprocess + tempfile (безопасное выполнение кода)
- AST (статическая проверка безопасности)
- FastAPI (REST API)
- Celery + RabbitMQ + Redis (асинхронная очередь задач)
- Streamlit (UI)

## Потенциальный data flow

Основной поток данных:

> Research Question → DecomposerAgent → Structured Plan → CodeGeneratorAgent → Code → TestingAgent → Test Results → [OptimizationAgent → Improved Code]* → FormatterAgent → Final Research Output

**Делегируется LLM:**
* формализация задачи
* декомпозиция
* вывод алгоритма
* генерация кода
* генерация тест-кейсов
* оптимизация кода
* принятие решений роутером (TEST / OPTIMIZE / FORMAT)

**Не делегируется LLM:**
* выполнение кода (subprocess)
* проверка безопасности (AST)
* управление workflow (очередь, retry)
* логирование и мониторинг
