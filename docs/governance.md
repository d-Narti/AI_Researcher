## Risk Register

| Риск | Вероятность | Влияние | Детект | Защита | Остаточный риск |
|------|-------------|---------|--------|--------|-----------------|
| Галлюцинации модели | medium | high | Автотесты (success_rate) | Декомпозиция + итеративная оптимизация | medium |
| Ошибки генерации кода | high | medium | Runtime ошибки в subprocess | Тестирование + retry цикл | medium |
| Неверная формализация задачи | medium | high | Manual review при low success_rate | Чёткие промпты с примерами | medium |
| Prompt injection | low | high | Prompt audit (ручной) | Фиксированные шаблоны, нет user-input в system prompt | low |
| Высокая стоимость | medium | medium | Мониторинг числа итераций | Ограничение max_steps, local Ollama | medium |
| Недоступность LLM API | medium | high | Retry с backoff (до 3 попыток) | Exponential backoff retry в Celery worker | low |
| Небезопасный сгенерированный код | low | high | AST-анализ до выполнения | Blocklist модулей (subprocess, socket, requests…) | low |

## Безопасность и данные

Система логирует:

* входную задачу (task)
* решения агентов (decomposition, code)
* результаты тестирования (test_results, success_rate)
* время выполнения каждого шага
* ошибки и исключения

Система **не обрабатывает персональные данные** по умолчанию — вход представляет собой научные задачи. При использовании в корпоративной среде необходимо провести отдельный аудит входных данных.

## Политика хранения логов

* **Срок хранения:** 31 день (PoC)
* **Формат:** текстовые строки (по умолчанию) или структурированный JSON (при `LOG_FORMAT=json`)
* **Хранение:** stdout → файл (PoC); в production — S3-compatible object storage или централизованный log aggregator (Loki, CloudWatch)
* **Доступ:** только разработчик системы (PoC); в production — роль Read-Only для on-call инженеров
* **Ротация:** не реализована в PoC; в production — ежедневная ротация с retention 31 дней

## Защита от injection

Меры защиты:

* Строгие system prompts с фиксированным форматом
* Пользовательский ввод (`task`) передаётся только в user-часть промпта, никогда в system prompt
* Статический AST-анализ сгенерированного кода перед выполнением
* Blocklist опасных модулей: `subprocess`, `socket`, `requests`, `urllib`, `os.system`, `shutil`, `ctypes`

## Подтверждение действий

В текущем PoC система **не запрашивает подтверждение** пользователя перед выполнением действий.

**Обоснование:**
* Выполнение кода происходит в изолированной среде (временный файл через `tempfile`, процесс через `subprocess`)
* AST-проверка блокирует потенциально опасные операции до выполнения
* Нет файловых операций вне sandbox, нет сетевых вызовов из сгенерированного кода
* Таймаут выполнения: 5 секунд на тест-кейс

**В production-версии** рекомендуется добавить:
* Подтверждение перед выполнением кода при первом запуске задачи
* Отображение сгенерированного кода пользователю с возможностью отмены

## Методы детекта

### Автоматические тесты

* Запускаются после каждой генерации/оптимизации кода
* Метрика: `success_rate` — доля пройденных тест-кейсов
* Алерт (логирование WARNING): при `success_rate = 0` и `total_tests > 0`

### Runtime monitoring

* Непрерывный мониторинг через Celery worker logs
* Все `stderr` сгенерированного кода записываются в `test_results.errors`
* Ошибки Celery worker логируются на уровне ERROR

### Prompt audit

* **Формат:** ручной
* **Кто:** разработчик системы
* **Периодичность:** раз в неделю (PoC) или при `success_rate < 50%` за скользящие 24 часа
* **Что проверяется:** случайная выборка 5–10 запросов: корректность декомпозиции, качество кода, соответствие тестов задаче

### Manual review

* **Когда:** при security-событиях (AST-блокировка кода), при трёх consecutive провалах подряд, при неожиданных runtime-ошибках в prod
* **Кто:** разработчик системы
* **Процесс:** просмотр полного pipeline по `task_id` в логах, ручное воспроизведение задачи

## Audit Trail

Каждый запрос фиксирует полный след выполнения:

```json
{
  "task_id": "celery-task-uuid",
  "task": "исходная задача пользователя",
  "steps": [
    {"type": "DECOMPOSE", "result": "..."},
    {"type": "GENERATE", "result": "..."},
    {"type": "TEST", "success_rate": 0.6},
    {"type": "OPTIMIZE", "result": "..."},
    {"type": "TEST", "success_rate": 1.0},
    {"type": "FORMAT", "result": "..."}
  ],
  "final_success_rate": 1.0,
  "status": "completed"
}
```

**Воспроизведение инцидента:**

1. Найти `task_id` в логах по времени инцидента: `grep "task_id" logs/ | grep "TIMESTAMP"`
2. Получить полный trace: `grep "task_id=<id>" logs/`
3. Просмотреть входную задачу, каждый шаг агента, решения роутера, итоговый результат
4. При необходимости — перезапустить задачу с теми же параметрами через API

**Логируемые поля при каждом шаге:**
* `task_id` (Celery task UUID)
* `step` (DECOMPOSE / GENERATE / TEST / OPTIMIZE / FORMAT)
* `agent` (имя агента)
* `success_rate` (для TEST-шагов)
* `error` (при наличии)
* `timestamp`
